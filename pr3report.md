## Контрольные вопросы с ответами

### 1. Что такое автоматизированное тестирование?

Автоматизированное тестирование — это способ проверки программного продукта с помощью специальных тестовых сценариев и инструментов без необходимости выполнять все проверки вручную. Такие тесты запускаются автоматически и позволяют быстро выявлять ошибки в коде.

### 2. Какие задачи решают автоматизированные тесты?

Автоматизированные тесты позволяют:

1. Проверять корректность работы программы.
2. Быстро находить ошибки после изменения кода.
3. Предотвращать повторное появление уже исправленных дефектов.
4. Ускорять проверку проекта перед выпуском новой версии.
5. Повышать качество и стабильность программного продукта.

### 3. Какие существуют виды автоматизированных тестов?

Основные виды автоматизированных тестов:

1. Unit-тесты.
2. Интеграционные тесты.
3. E2E-тесты.
4. Регрессионные тесты.
5. Нагрузочные тесты.
6. Smoke-тесты.

В рамках данной работы основное внимание уделяется unit-, integration- и E2E-тестам.

### 4. Что такое unit-тест?

Unit-тест — это тест, который проверяет отдельную небольшую часть программы: функцию, метод или модуль. Его задача — убедиться, что конкретный элемент работает правильно изолированно от остальных частей системы.

### 5. Как написать unit-тест?

Чтобы написать unit-тест, необходимо:

1. Выбрать функцию или метод для проверки.
2. Подготовить входные данные.
3. Вызвать тестируемую функцию.
4. Сравнить фактический результат с ожидаемым.
5. Зафиксировать успешное или ошибочное поведение.

В Go unit-тесты обычно пишутся в файлах с суффиксом `_test.go` с использованием встроенного пакета `testing`.

### 6. Что такое интеграционный тест?

Интеграционный тест проверяет взаимодействие нескольких компонентов системы. Например, работу HTTP-обработчика вместе с сервисом и базой данных. Такой тест показывает, правильно ли обмениваются данными части приложения.

### 7. Как провести интеграционное тестирование?

Интеграционное тестирование проводится следующим образом:

1. Определяются взаимодействующие компоненты.
2. Подготавливается тестовое окружение.
3. Запускаются реальные или приближенные к реальным зависимости.
4. Выполняются тестовые сценарии.
5. Анализируются результаты и возможные ошибки взаимодействия.

### 8. Что такое E2E-тест?

E2E-тест (End-to-End) — это тест, который проверяет работу всей системы целиком с точки зрения пользователя. Он имитирует полный сценарий использования приложения: от отправки запроса или открытия страницы до получения конечного результата.

### 9. Как выполнить E2E-тестирование?

Для выполнения E2E-тестирования необходимо:

1. Подготовить полностью работоспособное тестовое окружение.
2. Запустить приложение и все необходимые сервисы.
3. Смоделировать действия пользователя.
4. Проверить итоговый результат работы системы.
5. Зафиксировать успешное прохождение или ошибку.

### 10. Как интегрировать автоматизированные тесты в процесс CI/CD?

Автоматизированные тесты интегрируются в CI/CD следующим образом:

1. При каждом коммите или pull request автоматически запускается пайплайн.
2. На этапе CI выполняются сборка проекта, линтеры и тесты.
3. Сначала обычно запускаются unit-тесты как самые быстрые.
4. Затем запускаются интеграционные тесты.
5. После этого могут выполняться E2E-тесты.
6. Если тесты проходят успешно, проект может быть допущен к развёртыванию.
7. Если тесты не проходят, пайплайн останавливается и разработчик исправляет ошибки.

## Практическое задание

### План внедрения автоматизированных тестов в процесс разработки проекта

Для проекта на Go можно использовать многоуровневый подход к тестированию.

#### 1. Unit-тесты

Unit-тесты применяются для проверки:

1. Бизнес-логики.
2. Отдельных функций и методов.
3. Валидации входных данных.
4. Расчётов и преобразований данных.

Unit-тесты должны запускаться при каждом изменении кода, так как они выполняются быстро и позволяют сразу обнаружить ошибки в логике программы.

#### 2. Интеграционные тесты

Интеграционные тесты применяются для проверки:

1. Работы HTTP-обработчиков.
2. Взаимодействия сервиса с базой данных.
3. Обмена данными между несколькими модулями.
4. Корректности API-запросов и ответов.

Эти тесты позволяют убедиться, что отдельные части проекта правильно работают вместе.

#### 3. E2E-тесты

E2E-тесты применяются для проверки:

1. Основных пользовательских сценариев.
2. Полного пути прохождения запроса через систему.
3. Корректности работы приложения в среде, близкой к реальной.

Такие тесты обычно выполняются реже, чем unit-тесты, так как требуют больше времени и ресурсов.

### Интеграция тестов в CI/CD

В рамках CI/CD можно использовать следующий порядок:

1. Разработчик отправляет изменения в репозиторий.
2. GitHub Actions автоматически запускает пайплайн.
3. Выполняется установка зависимостей и сборка проекта.
4. Запускаются unit-тесты.
5. При успешном прохождении запускаются интеграционные тесты.
6. Далее запускаются E2E-тесты.
7. При успешном прохождении всех этапов изменения считаются готовыми к слиянию или развёртыванию.

Такой подход позволяет проверять код автоматически и снижает вероятность попадания ошибок в рабочую версию приложения.

## Примеры тестов на Go

### Пример unit-теста

Ниже приведён пример функции и unit-теста для проверки сложения двух чисел.

```go
package mathutils

func Sum(a, b int) int {
	return a + b
}
```

```go
package mathutils

import "testing"

func TestSum(t *testing.T) {
	result := Sum(2, 3)
	expected := 5

	if result != expected {
		t.Fatalf("expected %d, got %d", expected, result)
	}
}
```

### Пример интеграционного теста

Ниже приведён пример интеграционного теста для HTTP-обработчика на Go.

```go
package handlers

import (
	"net/http"
)

func HealthHandler(w http.ResponseWriter, r *http.Request) {
	w.WriteHeader(http.StatusOK)
	_, _ = w.Write([]byte("ok"))
}
```

```go
package handlers

import (
	"net/http"
	"net/http/httptest"
	"testing"
)

func TestHealthHandler(t *testing.T) {
	req := httptest.NewRequest(http.MethodGet, "/health", nil)
	rec := httptest.NewRecorder()

	HealthHandler(rec, req)

	if rec.Code != http.StatusOK {
		t.Fatalf("expected status %d, got %d", http.StatusOK, rec.Code)
	}

	if rec.Body.String() != "ok" {
		t.Fatalf("expected body %q, got %q", "ok", rec.Body.String())
	}
}
```

В данном примере проверяется взаимодействие HTTP-запроса, обработчика и HTTP-ответа.

### Пример E2E-теста

Ниже приведён упрощённый пример E2E-теста, который проверяет запуск тестового сервера и выполнение запроса к конечной точке приложения.

```go
package e2e

import (
	"io"
	"net/http"
	"net/http/httptest"
	"testing"
)

func TestE2EHealthEndpoint(t *testing.T) {
	handler := http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
		w.WriteHeader(http.StatusOK)
		_, _ = w.Write([]byte("service is running"))
	})

	server := httptest.NewServer(handler)
	defer server.Close()

	resp, err := http.Get(server.URL)
	if err != nil {
		t.Fatalf("request failed: %v", err)
	}
	defer resp.Body.Close()

	body, err := io.ReadAll(resp.Body)
	if err != nil {
		t.Fatalf("failed to read response body: %v", err)
	}

	if resp.StatusCode != http.StatusOK {
		t.Fatalf("expected status %d, got %d", http.StatusOK, resp.StatusCode)
	}

	if string(body) != "service is running" {
		t.Fatalf("unexpected body: %s", string(body))
	}
}
```

В этом примере тестируется полный пользовательский сценарий: запуск сервера, выполнение HTTP-запроса и проверка ответа.

## Пример интеграции в CI/CD

Ниже приведён пример конфигурации `GitHub Actions` для автоматического запуска тестов.

```yaml
name: Go CI

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout repository
        uses: actions/checkout@v4

      - name: Setup Go
        uses: actions/setup-go@v5
        with:
          go-version: '1.22'

      - name: Install dependencies
        run: go mod tidy

      - name: Run unit tests
        run: go test ./... -run TestSum

      - name: Run integration tests
        run: go test ./... -run TestHealthHandler

      - name: Run E2E tests
        run: go test ./... -run TestE2EHealthEndpoint
```

Данный пайплайн автоматически запускает тесты при отправке изменений в репозиторий и при создании pull request.
